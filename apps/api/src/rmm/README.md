# RMM Module (Tactical RMM)

- `rmm.module.ts` — leaf module exporting `RmmService`, the Tactical RMM REST client (`X-API-KEY` auth), plus `RmmController`:
  - `GET /rmm/agents` and `GET /rmm/agents/:agentId` — read-only device inventory, gated on **membership** of the configured alert channel.
  - `GET /rmm/agents/:agentId/remote-control` — mediates a live MeshCentral "Take Control" session via Tactical RMM's own `GET /agents/{id}/meshcentral/` endpoint (same `X-API-KEY`, no separate MeshCentral credentials needed). Gated on being an **`ADMIN`** of that channel (`ChannelsService.isChannelAdmin`), a stricter check than plain membership. There is currently no self-service way to promote a `ChannelMember` to `ADMIN` — a deployment bootstraps its first RMM operator by setting that role directly (e.g. via `npm run prisma:studio`), same as the e2e test does. A promotion endpoint is future work (`docs/ideias-futuras.md` §2, "Painel de administração").
- `rmm-webhook.module.ts` — top-level module exposing `POST /webhooks/rmm/alerts`, which receives Tactical RMM's outbound Alert Template webhook, posts a `SYSTEM` message into the configured channel (or opens a GLPI ticket via the existing `/ticket` flow for alerts at or above `RMM_AUTO_TICKET_SEVERITY`), and broadcasts it over the existing chat socket — mirroring `glpi-webhook.module.ts`.

**The remote-control URL is a live, bearer-equivalent credential** (it carries a one-time MeshCentral login token) — the controller returns it only in the direct HTTP response to the requesting user, and it is never persisted, never posted to a chat message, never broadcast to a room. `apps/web`'s `RmmAgentsPanel` fetches it fresh right before `window.open`-ing it and never caches it. Every issuance is recorded via `AuditService` (`apps/api/src/audit/`, action `rmm.remote_control.requested`) — this only proves a session was *granted*, not what happened inside it, since that's MeshCentral's own session log, outside Elyzian's visibility.

Tactical RMM's webhook body is a JSON template you author yourself in its Alert Template UI (using `{{ }}` variables like `{{agent.hostname}}`, `{{alert.message}}`), so the receiving contract here — `alertId`, `hostname`, `client`, `site`, `severity`, `message`, `resolved` (`packages/shared/src/rmm.dto.ts`) — is one Elyzian defines; the Alert Template must be configured to emit exactly that shape. Unlike GLPI's HMAC-signed webhook, Tactical RMM can't sign the body, so `RMM_WEBHOOK_SECRET` is checked as a static `Authorization: Bearer <token>` header instead.

Alert-authored messages are posted by a fixed, seeded system user (`username: rmm-bot`, migration `20260716140000_add_rmm_alert_ref_and_system_user`) rather than any real person — `MessagesService.createSystemMessage()` is generic enough for future bot integrations to reuse the same account.

Resolved/unresolved state is tracked in `RmmAlertRef` (keyed on Tactical RMM's `alertId`, mirroring how `TicketRef` tracks GLPI ticket status) so a later "resolved" webhook can find and update the original chat message instead of posting a new one.

Configured via `RMM_URL`, `RMM_API_KEY`, `RMM_WEBHOOK_SECRET` (optional), `RMM_ALERT_CHANNEL_NAME`, `RMM_AUTO_TICKET_SEVERITY` — see `.env.example`.

**Scope note:** receiving alerts, listing agents, and mediating interactive MeshCentral remote-desktop sessions are wired up. Tactical RMM's script/command endpoints (`POST /agents/{id}/runscript/`, `POST /agents/{id}/cmd/`), the mesh-agent `recover/` endpoint, and the separate `WebVNC` endpoint are intentionally **not** wired up — running arbitrary code is a different risk profile than launching an interactive session a human directly controls, and would need its own allowlist/access-control design.
